# /deliverables - Client-Facing Documentation

This folder contains documentation meant for the client, not internal development.

---

## Files

| File | Purpose | Audience |
|------|---------|----------|
| [ROLLOUT.md](ROLLOUT.md) | Progress tracker (powers /rollout page) | Public |
| [PROGRESS.md](PROGRESS.md) | Detailed completed work log | Client |
| [CHANGELOG.md](CHANGELOG.md) | What's new in each release | Client |
| [TRAINING.md](TRAINING.md) | How to use the admin/system | Client |
| [CLIENT_UPDATE.txt](CLIENT_UPDATE.txt) | Latest message to send | Developer reference |

---

## ROLLOUT.md

**Purpose:** High-level progress tracker shown on the `/rollout` page.

**Updates:** After completing significant milestones, run `/update-rollout`.

**Visible to:** Anyone with the URL (no auth required).

---

## PROGRESS.md

**Purpose:** Detailed log of what's been completed.

**Updates:** After each work session or feature completion.

**Format:**
```markdown
## [Date]

### Completed
- Feature X implemented
- Bug Y fixed

### Notes
- Decision made about Z
```

---

## CHANGELOG.md

**Purpose:** User-friendly release notes.

**Updates:** Before each deployment or major release.

**Format:**
```markdown
## v1.0.0 - [Date]

### Added
- New feature X

### Fixed
- Bug in Y

### Changed
- Behavior of Z
```

---

## TRAINING.md

**Purpose:** How-to guide for the client to use the system.

**Updates:** When UI/workflow changes significantly.

**Should cover:**
- How to log in
- How to manage content
- How to view analytics
- Common tasks step-by-step

## Workflow

1. **Do work** on the project
2. **Run `/update-rollout`** to sync documentation
3. **Run `/update-client`** to generate message
4. **Send message** to client
5. **Repeat**
